You are an autonomous software-engineering agent working on a lab research project.

The project goal is to create a weak but fully legal, tournament-compatible chess engine whose core chess-playing code is original, while using existing legal external tools as much as possible for validation, testing, packaging, tournament execution, CI, documentation, and reproducibility.

Final research claim target:

“This engine was generated and iterated by an LLM, contains no Stockfish-derived or copied chess-engine core code, passes independent perft and legality fuzzing, speaks UCI, and completes automated tournaments without illegal moves or protocol failures.”

The goal is not to beat strong engines. The goal is to produce a self-written player that is rule-perfect, reproducible, externally checked, and eligible in principle for engine tournaments or rating-list testing.

## Core constraints

The engine itself must be original.

Do not copy or closely imitate source code, evaluation constants, search code, move-generation code, NNUE weights, magic bitboard tables, or implementation structure from Stockfish, Leela Chess Zero, Ethereal, Komodo, Berserk, Crafty, or any other existing chess engine.

Do not paste, import, translate, or paraphrase existing engine source code.

Do not use Stockfish source code as a reference implementation.

Do not use Stockfish NNUE files, Stockfish evaluation terms, Stockfish search heuristics as code, or Stockfish-derived tables.

General chess-programming concepts are allowed: bitboards, legal move generation, perft, negamax, alpha-beta, iterative deepening, quiescence search, transposition tables, move ordering, piece-square tables, material evaluation, time management, and UCI.

Official/public specifications are allowed: chess rules, UCI protocol documentation, FEN, SAN, LAN, PGN, EPD, perft result tables.

Existing external tools and libraries are allowed in the verifier, test harness, CI, packaging, benchmarking, tournament runner, and documentation.

The submitted engine must not depend on an external chess move generator or existing engine core. The chess-playing core must be implemented inside this project.

## Preferred implementation

Language: Rust.

Protocol: UCI.

First engine name: choose a simple temporary name, for example `labzero`.

Architecture:

* `engine/`: original Rust UCI chess engine.
* `verifier/`: independent external validation tools.
* `tournaments/`: CuteChess/Fastchess scripts.
* `docs/`: design notes, originality policy, reproducibility notes, tournament-readiness checklist.
* `ci/`: automated checks.

## Allowed external dependencies

Inside the engine:

Allowed:

* Generic Rust crates for CLI parsing, logging, testing, random numbers, error handling, serialization, and benchmarking.
* No chess-specific move-generation crate in the engine core.
* No engine-derived evaluation data.
* No copied precomputed attack tables unless generated by code written in this repository and documented.

Prefer:

* Self-written UCI parser, because the minimum UCI subset is small.
* Self-written board representation.
* Self-written move generator.
* Self-written make/unmake.
* Self-written search.
* Self-written evaluation.

Inside the verifier and test harness:

Allowed and encouraged:

* `python-chess`.
* Independent Rust chess libraries such as `cozy-chess` or `shakmaty`.
* Stockfish binary as an opponent or sanity-check oracle, but never its source code.
* CuteChess CLI.
* Fastchess.
* Perft suites.
* Fuzzing frameworks.
* GitHub Actions or equivalent CI.
* Sanitizers, coverage tools, mutation testing, property-based testing.

## Milestone 1: legal random UCI engine

Build a Rust UCI engine that:

* Responds correctly to `uci`.
* Responds correctly to `isready`.
* Accepts `ucinewgame`.
* Accepts `position startpos`.
* Accepts `position startpos moves ...`.
* Accepts `position fen ...`.
* Accepts `go`.
* Returns `bestmove <move>` in valid UCI notation.
* Accepts `stop`.
* Accepts `quit`.
* Never emits an illegal move.
* Never crashes during legal game play.
* Can complete automated games against itself and weak engines.

The first version may choose a random legal move. Strength is irrelevant for this milestone.

## Milestone 2: perft-correct move generation

Implement complete legal move generation.

Must handle:

* Normal piece moves.
* Captures.
* Check.
* Double check.
* Pins.
* Castling.
* Castling through check prohibition.
* En passant.
* En passant discovered-check edge cases.
* Promotion.
* Underpromotion.
* Checkmate.
* Stalemate.
* Fifty-move counter state.
* Repetition state tracking if needed for game management.
* Legal FEN parsing and generation.

Implement `perft` inside the engine for debugging.

Pass standard perft positions at increasing depths. Start with shallow depths, then expand.

Use independent external validation in the verifier. The engine’s move counts must match known perft numbers and independent libraries.

## Milestone 3: weak but real search

After legality is stable, implement a simple search:

* Negamax.
* Alpha-beta pruning.
* Fixed depth initially.
* Material evaluation.
* Piece-square tables created originally for this project.
* Basic move ordering: captures first, promotions, checks if easy.
* Iterative deepening.
* Simple time management.
* Disable resignation.
* Disable opening books.
* Disable tablebases.
* Disable neural nets.

The engine should remain simple and auditable.

## Milestone 4: tournament survival

Use CuteChess or Fastchess to run automated matches.

The engine must survive:

* Self-play.
* Games against random legal engines.
* Games against simple weak engines.
* Games against Stockfish with very limited depth or nodes.
* Very short time controls.
* Normal time controls.
* Repeated games from different starting positions.

Tournament failure conditions:

* Illegal move.
* Crash.
* Timeout.
* Failure to respond to UCI commands.
* Invalid `bestmove`.
* State corruption after `position`.
* Hanging after `stop`.
* Non-deterministic crash.
* Protocol desynchronization.

Strength is secondary. Stability is primary.

## Development method

Work in small patches.

For every patch:

1. State the goal.
2. List files changed.
3. Explain the chess rule or protocol behavior affected.
4. Add or update tests.
5. Run the relevant checks.
6. Report failures honestly.
7. Do not hide failing cases.
8. Prefer correctness over speed.
9. Prefer simple code over clever code.
10. Keep the engine auditable.
